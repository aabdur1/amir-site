- **Method:** `security-review` skill, adapted from diff-review to full-repo audit — one

## Verdict: No exploitable vulnerabilities found (high confidence)

Zero findings survive at confidence ≥ 8.

This is an **expected** result, not a surprising one. The site is statically rendered with
no server-side user-input processing: no `app/api`, no `route.ts` handlers, no
`middleware.ts`, no forms posting to a backend. The classic injection / authz / authn
attack surface essentially does not exist server-side, and every dynamic-looking sink
traces back to hardcoded, author-authored, or integrity-checked data.

## Surfaces examined and cleared

### 1. `dangerouslySetInnerHTML` — 6 usages · verified 9/10
- `app/layout.tsx:65` (theme script) — hardcoded string literal, zero interpolation.
- `app/layout.tsx:71` (`Person` JSON-LD) — fully hardcoded object literal. **Credly data is
  NOT present here.**
- `app/learn/[slug]/page.tsx:117,121` and `app/work/[slug]/page.tsx:87,91` — `JSON.stringify`
  of objects fed only by post-`notFound()` values from the static arrays
  `lib/learn/artifacts.ts` / `lib/work/case-studies.ts`.
- The only dynamic value (`slug`) reaches these sinks only *after* `getArtifact(slug)` /
  `getCaseStudy(slug)` succeeds; unknown slugs hit `notFound()` (throws) first, so only the
  fixed set of known slugs can ever be reflected.

### 2. Credly API (only external runtime fetch) · verified 9/10
- `lib/badges.ts:94` fetches `credly.com/.../badges.json` at build/revalidate.
- `getAllBadges()` is consumed in exactly one place (`app/page.tsx` → `<Certifications>`); it
  never reaches metadata or any JSON-LD.
- Every field renders through auto-escaping React JSX (`name` → alt/aria-label,
  `shortName`/`org` → text nodes) or `next/image` `src`, host-restricted by `remotePatterns`
  to `images.credly.com` and re-constrained by CSP `img-src`. `dangerouslyAllowSVG` is unset
  (default false), so a remote SVG payload is refused by the optimizer.
- Badge `url` is *constructed* (`https://www.credly.com/badges/${id}`), not taken raw — no
  `javascript:` scheme reachable even if `id` were hostile.

### 3. webR download pipeline (supply chain) · verified 8/10 (adversarial)
This is the crown-jewel risk: downloaded WASM binaries are served first-party and executed in
visitors' browsers, and the `/webr/:path*` route serves them with a CSP containing
`'unsafe-eval'`. A supply-chain compromise here would be high severity.

- The sha256 lock (`scripts/webr-assets.lock.json`) is **fail-closed**. On a fresh CI build,
  `public/webr/` is gitignored/absent → `verifyLock()` returns false → `populate()`
  `rm -rf`s then re-downloads → `recordOrCheckLock()` runs a strict bijection check:
  `mismatches` (locked file whose fresh hash differs or is missing) ∪ `extra` (downloaded file
  not in lock); either non-empty → `process.exit(1)`.
- `package.json` wires `prebuild` to `fetch-webr.mjs` with **no `--update`**, so the check
  branch (not the record branch) runs in CI. A failing `prebuild` halts `npm run build` → no
  deploy. Fail-closed at every layer.
- HTTPS-only (`github.com`, `repo.r-wasm.org`). Redirects are followed, but every byte is
  hash-checked against the committed lock, so a malicious redirect target fails.
- All 16 `.tgz` binaries + the `PACKAGES` index are locked. `PACKAGES.gz` is lock-exempt but
  is generated locally via `gzipSync` from the same in-memory bytes that become the *locked*
  `PACKAGES` — no MITM vector. A tampered `PACKAGES` can only cause different files/versions
  to be fetched, which surface as `extra`/`mismatch`.
- Verifier reproduced the exit-1 path empirically (corrupted a locked hash → script detected
  divergence, re-downloaded, printed `LOCK MISMATCH`, exited 1).

### 4. WASM sandboxes running user code (sql.js / Pyodide / webR)
- All execute in the visitor's own browser (sql.js/Pyodide on main thread, webR in a Web
  Worker). No server component, no cross-user data, no persistence beyond the user's own
  `localStorage`. A user tampering with the checker only fools their own progress bar.
- R user code travels as a bound R variable (`globalEnv.bind('.webr_user_code', ...)`), not
  string-interpolated — no eval-injection even in principle.
- Falls under the client-code-is-untrusted-by-design exclusion.

### 5. Ops scripts (`add-photo.mjs`, `add-photo-gui.mjs`, `fetch-webr.mjs`)
- Use `execFileSync` (argv, no shell) for `aws`/`osascript`/`node`, so filenames can't inject
  commands. No untrusted-input path (owner runs them on their own files).
- S3 keys use `path.basename`, neutralizing traversal.

### 6. Tableau embed (`components/work/tableau-embed.tsx`)
- Click-gated plain iframe; `embed.url` hardcoded in `case-studies.ts`; CSP `frame-src` scoped
  to `public.tableau.com` on that one route; no third-party script enters page context.
- Site protected by `X-Frame-Options: DENY` + `frame-ancestors 'none'`.

### 7. `photos.json`, route slugs, OG image routes · verified 9/10
- `public/photos.json` is a committed repo file (static local import), not externally fetched;
  camera/lens/date render as escaped JSX; `url`/`thumb` → allowlisted CloudFront `Image` src.
- `/learn/[slug]` and `/work/[slug]` gate unknown slugs behind `notFound()` before any
  reflection.
- OG image routes (`opengraph-image.tsx`) deliberately do NOT call `notFound()`, but their
  output is a **rasterized PNG** (satori/resvg) — raw slug becomes pixels, never reaching
  HTML/JS/JSON-LD. (An arbitrary slug 200s a generic OG card — cosmetic content-spoofing only,
  not XSS.)

### 8. Secrets
- No tracked `.env*` or credential files; `.env*` and `/public/webr/` gitignored; no AWS keys,
  private keys, or API tokens found in tracked files.

### 9. CSP / security headers (`next.config.ts`, `netlify.toml`)
- `'unsafe-inline'` in `script-src` is documented and load-bearing (App Router inline RSC
  flight scripts) — but with no reflected/stored injection point it grants no exploit.
- `'unsafe-eval'` is added only on `/webr/:path*` asset responses (never a document), scoped
  to the worker's own dlopen path.
- Reasonable posture for this app.

## Defense-in-depth notes — NOT vulnerabilities, no action required

Both adversarial verifiers independently flagged these as latent footguns, not live issues.
Recorded so a future rerun can confirm they're still latent (and so they don't get
re-flagged as new).

1. **JSON-LD `</script>` breakout is currently unreachable but discipline-dependent.**
   `JSON.stringify` does not escape `<`, `>`, `/`. Safe today only because every value in
   those objects is hardcoded. If a future edit ever pipes Credly-fetched or user-supplied
   text into the JSON-LD (e.g. an auto-generated credentials `ItemList`), it becomes a stored
   XSS on an inline `<script type="application/ld+json">`. Cheap proactive guard:
   `.replace(/</g, '\\u003c')` on the stringified output.

2. **`verifyLock()` does not detect *extra* unlocked files** (only that locked files match).
   Exploiting requires a malicious file to already sit in gitignored `public/webr/` *before*
   the build — which a fresh Netlify build never does (it wipes and re-downloads), and
   unlocked files have no load path into the browser anyway. Strictly a build-cache-poisoning
   concern above the MITM threat model.

## Exclusions applied (per skill)

DoS / resource exhaustion / rate limiting; secrets-on-disk-if-secured; missing hardening
without a concrete exploit; outdated-dependency CVEs; theoretical race/timing; test-only and
documentation files; log spoofing; path-only SSRF; React/JSX XSS except via unsafe methods;
client-side lack of authz (untrusted by design); shell command-injection without an
untrusted-input path; subtle web issues (tabnabbing, prototype pollution, open redirect)
unless extremely high confidence.
